# Sales Pipeline Case — Project Reference

## Purpose

Interview case study demonstrating Power BI skills: star schema data modelling, DAX measure authoring, and PBIP-format project management. The dataset is a synthetic B2B sales pipeline for a DACH-region company (150 deals, snapshot data, no stage history).

## Data Model

### Star Schema

```
dim_sales_rep ──────┐
dim_lead_source ────┤
dim_industry ───────┼──→ fct_sales_pipeline ←── dim_stage_definitions
dim_country ────────┤
dim_customer ───────┘
```

All dimension tables are M-query based, pulling distinct values from the same Excel source as the fact table. Relationships are all many-to-one (fact → dim), single-direction filter propagation.

### Fact Table: `fct_sales_pipeline`

Source: `Sales_Pipeline.xlsx` → Sheet: `Sales Pipeline`, loaded via `Web.Contents` from GitHub raw URL.

| Column | Type | Notes |
|---|---|---|
| Lead ID | Text | Primary key |
| Customer Name | Text | FK → dim_customer |
| Lead Source | Text | FK → dim_lead_source |
| Sales Rep | Text | FK → dim_sales_rep |
| Stage | Text | FK → dim_stage_definitions |
| Stage Change Date | Date | Date of last stage transition |
| Expected Close Date | Date | Projected close date |
| Expected Order Volume (€) | Integer | Primary monetary measure |
| Probability (%) | Decimal | Deterministic by stage (not independent signal) |
| Loss Reason | Text | Populated only when Stage = Lost |
| Industry | Text | FK → dim_industry |
| Country | Text | FK → dim_country |
| Contact Person | Text | No analytical value |
| Opportunity Name | Text | No analytical value |
| Is Open | Boolean | Calculated column: Stage NOT IN {Won, Lost} |
| Is Closed | Boolean | Calculated column: Stage IN {Won, Lost} |

### Dimension: `dim_stage_definitions`

Source: `Sales_Pipeline.xlsx` → Sheet: `Definitions`. Contains stage metadata (ID, Stages, Probability). Has a `Sort Order` calculated column (1–6) used to sort stages correctly in visuals. The `Stages` column is set to `sortByColumn: Sort Order`.

## Measures (`_measuretable.tmdl`)

All measures live in `_measuretable`. Organized by `displayFolder`:

### Core KPIs
| Measure | Description |
|---|---|
| `Total Pipeline Value` | SUM of volume for open deals only (NOT Won/Lost) |
| `Weighted Pipeline Value` | SUMX open deals: volume × probability |
| `Total Won Revenue` | SUM of volume where Stage = Won |
| `Total Lost Value` | SUM of volume where Stage = Lost |
| `Open Deals` | COUNTROWS where Stage NOT IN {Won, Lost} |
| `Win Rate` | DIVIDE(Won, Won+Lost) — closed deals only in denominator |
| `Avg Deal Size` | AVERAGE of Expected Order Volume across all deals |

### Rep Performance
| Measure | Description |
|---|---|
| `Deals Won` | COUNTROWS where Stage = Won |
| `Deals Lost` | COUNTROWS where Stage = Lost |
| `Deal Count` | COUNTROWS — all deals in current filter context |

### Stage Analysis
| Measure | Description |
|---|---|
| `Stage Deal Count` | COUNTROWS per stage (implicit via visual filter context) |
| `Stage Value` | SUM of volume per stage |
| `Stage Weighted Value` | SUMX volume × probability per stage |

### Loss Analysis
| Measure | Description |
|---|---|
| `Loss Count` | COUNTROWS where Stage = Lost, sliceable by Loss Reason |
| `Loss Rate by Source` | DIVIDE(lost, ALL(Stage) total) — uses ALL(Stage) for correct denominator |

## TMDL Ground Rules

These rules are critical. Violating them causes Power BI to either reject the file entirely or silently overwrite your changes.

### 1. `///` doc comments are only valid on tables and measures
`///` maps to the TMDL `description` property. **Relationships do not have a description property** — placing `///` above a relationship block causes a hard parse error on open (`Property 'description' is unknown`). Tables and measures support it.

### 2. Never write TMDL files while Power BI Desktop has the project open
Power BI Desktop holds the model state in memory and auto-saves to disk. Any TMDL file you write externally while PBI is open will be overwritten when PBI saves — silently, without warning. Always close PBI Desktop before editing TMDL files externally.

### 3. Every new table must be registered in `model.tmdl`
Adding a `.tmdl` file in the `tables/` folder is not enough. Power BI will ignore it unless it is also listed as `ref table <tablename>` in `model.tmdl`. The table name in the `ref table` line must match the `table` declaration in the `.tmdl` file exactly.

### 4. Every new table and relationship must also be in the model diagram
After adding new tables and opening PBI, the Model view may not auto-position new tables. Arrange them manually, then save — Power BI writes the positions to `diagramLayout.json`.

### 5. Relationship IDs must be unique strings
Existing auto-generated relationships use GUIDs (`4224b749-...`). Custom relationships can use readable IDs (`r-dim-sales-rep-01`) as long as they are unique within `relationships.tmdl`. Do not reuse IDs.

### 6. After external edits, always commit before reopening PBI
The correct workflow is: close PBI → edit TMDL files → commit to git → open PBI. This ensures the committed state matches what PBI loads, and prevents PBI's auto-save from overwriting uncommitted work.

### 7. `sortByColumn` requires the referenced column to exist in the same table
If you set `sortByColumn: 'Sort Order'` on a column, the `Sort Order` column must exist in the same table. If the sort column should be hidden from report view, add `isHidden` to it.

### 8. Calculated columns use `= expression` syntax; source columns use `sourceColumn:`
```tmdl
// Source column (from query):
column 'Lead Source'
    dataType: string
    sourceColumn: Lead Source

// Calculated column (DAX):
column 'Is Open' = NOT ( fct_sales_pipeline[Stage] IN { "Won", "Lost" } )
    dataType: boolean
```

## Key Limitations

- **No date dimension** suitable for time intelligence — only Stage Change Date and Expected Close Date exist, both as snapshot dates with no continuous history.
- **Probability is not independent** — it is deterministically derived from Stage. Do not use it as a separate analytical axis.
- **Only 3 customers** — customer-level slicing produces very thin cross-sections.
- **n=150** — any Rep × Industry × Country cross-filter can produce single-digit cells.
